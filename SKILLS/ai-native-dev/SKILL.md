# AI 原生开发法（认知容量约束开发）

> 版本: v1.1.0 | 修正：每个模块独立文件，头部自描述

---

## 触发条件

写新代码、新建模块、组织文件结构时，读取本技能。

---

## 核心理念

**认知容量是硬上限。** 一个人装不下百万行代码，一个 LLM 的有效上下文也撑不住几十个文件来回跳。所以：

> 每个模块一个独立的 `.py` 文件。头部注释不依赖项目上下文就能理解——读这个文件的人不需要先读 README、不需要知道其他模块在干嘛。就这个文件本身，自描述。

---

## 文件组织规则

### 1. 每个模块一个独立 `.py`

一个黑盒 = 一个文件。不把多个无关模块塞进一个文件。

### 2. 头部注释自描述

读这个文件的人只需要知道：给我什么 → 我还你什么。
不需要知道项目叫什么、谁写的、其他文件怎么实现的。

**坏的头部**（依赖项目上下文）：
```
"本模块负责 Mio 项目的记忆提取管线第三步"
```

**好的头部**（自描述）：
```
"接收一段 LLM 回复文本，从中提取用户偏好碎片。
 消费：对话文本字符串。
 产出：偏好碎片列表，每条含 {category, content, confidence}。"
```

### 3. 头部模板

```python
"""
# 模块名 (filename.py)

## 我是谁
一句话说清这个模块做什么。不引用项目名、不引用其他模块名。
读这行就知道这个文件在整个系统里的角色。

## 我消费什么
- 输入 A: 格式 + 示例值
- 输入 B: 格式 + 示例值

## 我产出什么
- 输出 A: 格式 + 示例值

## 我不需要知道
- 上游怎么实现的
- 下游怎么消费的
"""
```

### 4. 文件按数据流排列

不按 services/utils/schemas 分类。沿着数据流给文件和目录命名。
AI agent 读每个文件的"消费/产出"就能沿着链路找到上下游。

## 认知容量约束

| 约束 | 阈值 | 理由 |
|------|------|------|
| 同一时刻操作模块数 | ≤ 3 | 工作记忆上限 ~4 块 |
| 模块公共接口 | ≤ 5 个函数 | 深模块胜过浅模块 |
| 文件行数 | ≤ 300（建议） | 超过则考虑内部是否该再拆 |
| 模块间耦合 | < 0.5 | 改 A 必须改 B → 边界画错了 |
| 函数参数 | ≤ 3 | 超过 3 个参数难以记住用途 |
| 嵌套条件 | ≤ 2 层 | 深层嵌套超出认知追踪能力 |

### 冻结认知单元

模块稳定后不再改 → 变成"冻结构件" → 记住合约即可，不需要记住内部实现 → 释放认知容量给还在演化的模块。

---

## 与传统做法的对比

| 维度 | 传统 | AI 原生 |
|------|------|---------|
| 组织原则 | 按类型（services/utils/schemas） | 按数据流，每个模块独立文件 |
| 文档 | 另写 README | 文件头顶即文档，自描述 |
| 拆分 | SOLID 原则 | 每个独立模块一个 .py |
| 头部注释 | 可有可无 | 必须——不读项目也能看懂 |
| 目标读者 | 其他程序员（知道项目上下文） | AI agent + 不熟悉项目的人 |

---

## 检查清单

写代码前自问：
- [ ] 这个模块是独立的 `.py` 文件吗？
- [ ] 头部注释不依赖项目上下文能看懂吗？陌生人打开这个文件能理解它做什么吗？
- [ ] AI agent 能通过读"我消费/我产出"顺着链路找到上下游吗？
- [ ] 改这个模块需要同时改另一个模块吗？（如果是 → 耦合问题）
- [ ] 这个模块稳定后，能变成"冻结构件"不再占用认知容量吗？
