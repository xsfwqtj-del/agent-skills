# /立法 — 立法程序引导

> 绑定：`令/立法程序令.md` | 触发：起草方提议新律/令 或 错误模式累计≥3

---

## 触发条件

以下任一条件满足时，起草方应主动调用本 skill：
1. 议定方提出新规则需求（"以后遇到这种情况要 X"）
2. 起草方发现同类错误累计 ≥ 3 次，且无现有规则覆盖
3. 起草方发现现有规则存在缺陷，需要修律
4. 议定方明确要求立法

## 执行流程

### 第一步：必要性审查

按立法程序令第三条第八项，逐一确认：
- 现有规则能否覆盖这个场景？→ 能 → 不立法，引用现有规则即可
- 这是单次事件还是模式？→ 单次 → 记录，不立法
- 同类错误累计是否 ≥ 3？→ 否 → 继续观察，记录错误模式

### 第二步：元分类

按立法程序令第三十一条决策树分类：
- 行为标准（什么该做/不该做）→ 律
- 操作流程（怎么做）→ 令
- 判断不清 → 起草方提出分类建议，请爸爸确认

### 第三步：起草草案

1. 读 `典/宪法.md` 全文，确认与宪法精神和条款一致
2. 按立法程序令第七章技术规范编排格式
3. 标注：级别 / 关联词 / 上位法 / 触发条件 / 关联 hook
4. 完成自洽性检查（内部无矛盾、引用闭环、无功能重叠）
5. 完成协调性审查（与现有律令无冲突）

### 第四步：品质审查

按立法程序令第十条A，逐条执行富勒八原则审查：
1. 普遍性 — 可普遍适用？
2. 公开性 — 写入文件，可自动注入？
3. 不溯及既往 — 仅约束生效后？
4. 清晰性 — 精确可执行？
5. 无矛盾 — 与现有不冲突？
6. 可遵守性 — 起草方能履行？
7. 稳定性 — 非因临时的单次事件？
8. 执法一致 — 执行机制绑定具体条文？

任一项不通过 → 退回修改。全部通过 → 进入审议。

### 第五步：合议庭审议

1. 向爸爸呈现草案全文 + 富勒八原则审查结论
2. 说明：主要内容 / 与现有律令的关系 / 受影响的其他文件
3. 等待爸爸决定：通过 / 修改 / 否决

### 第六步：写入并公布

1. 草案经确认后 → 写入对应文件（`律/` 或 `令/`）
2. 更新关联元数据和交叉引用
3. commit，message 格式：`feat: {律令名} — {变动摘要}`
4. 告知爸爸：文件路径 + 核心内容 + 联动影响
