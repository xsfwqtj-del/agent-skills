# 技术方案写作流程

> 版本: v3.12.0 | 更新: 信息流补漏（步骤2→4消费）、补搜类型前置标注、内部文档判定、术语统一（官方/社区/个人）、搜索策略去代码化

---

## 交付标准

**完整七步流程**的方案必须同时满足以下五项。任何一项不达标，回到步骤 4 重写。

1. **树状可执行**：每个分支有独立的 ✅ 终点，一个不熟悉该领域的同行读完后能判断完成与否
2. **来源可追溯**：每条关键结论在终稿中保留检索关键词、来源 URL、来源质量标注
3. **依赖可验证**：分支间依赖关系标注正确，执行顺序无循环依赖
4. **自审已闭环**：审查日志记录了审查过程和结论。如有修正项则列出；若无修正项，写明审查依据说明为何无需修改
5. **边界已声明**：明确列出方案不覆盖的场景

**轻量流程**的交付标准仅为两项：结论有来源、步骤可执行。不要求树状图、审查日志和边界声明。

## 触发判断

**意图优先**。先判断用户是否需要「可执行的多步骤行动计划」，再参考关键词。语义不明时才用关键词辅助。

| 场景 | 判断依据 | 输出 | 流程 |
| :--- | :--- | :--- | :--- |
| "JWT 迁移方案""日志平台怎么搭" | ≥2 个独立决策 | plan.md 文档 | 完整七步 |
| "帮我把这个功能拆一下步骤" | 1 个，用户已告知技术选择，只需拆执行步骤 | 步骤列表 | 轻量（1→3快搜→4→7） |
| "这个函数什么意思""这段代码帮我改个变量名" | 无需决策 | 即时回答 | 直接回答 |

**关键词辅助**（语义不明时参考）：

触发词：写方案、出计划、迁移方案、改造方案、重构方案、架构设计、
          技术选型、怎么搭建、怎么部署、梳理思路、定方案、选哪个、选哪种

反触发词（出现时降低触发意愿）：
  "这行/这段/这里" + 修改 → 定点修复
  "什么意思/怎么理解/解释一下" → 纯答疑
  "怎么用这个 API" → 查文档

## 命名规则

方案文件命名遵循统一规则，在各步骤首次需要时取定：

{name} 取值优先级：① 用户明确给出的任务标题 → ② 任务中首个核心名词/动词组合（如"JWT 迁移""光伏电站规划"）→ ③ 仍无法确定时，向用户确认一次。统一转为 kebab-case（小写英文 + 连字符，中文部分用拼音或保留关键词）。

示例："JWT 认证迁移" → `jwt-auth-migration`；"数据库慢查询优化" → `slow-query-optimization`。

## 状态标签字典

以下为全文统一使用的状态标签，每个标签有明确的产生步骤和消费步骤。

| 标签 | 含义 | 产生于 | 消费于 |
|:---|:---|:---|:---|
| `[待核实]` | 不确定现状是否存在此问题 | 步骤 2 | 步骤 3 优先核实 |
| `[已确认存在]` | 已验证该问题确实存在 | 步骤 3 | 步骤 4 写入 |
| `[已确认不存在]` | 已验证该问题不存在，分支移除 | 步骤 3 | 步骤 4 不写入 |
| `[无直接参考]` | 两次降维搜索均无果 | 步骤 6 | 步骤 7 写入分支描述 |
| `[试验性方案]` | 基于推理而非外部参考的方案 | 步骤 6 | 步骤 7 写入分支描述 |

## 完整七步流程

```
1. 分析问题   了解现状和约束，理解根因（暂存上下文）
      ↓
2. 解构问题   拆分支、标依赖、关键路径+存量检查
      ↓
3. 搜索资料   核心信息搜索 + 发散联想 + 降维（暂存上下文）
      ↓   ⚠️ 优先核实 [待核实] 分支；仅当步骤1+3仍无法确认时，暂停询问用户
4. 写初稿     创建文件（若无文件系统则暂存内存），将步骤1-3落盘
      ↓   ⚠️ 步骤4未完成（初稿缺失任何必要章节）时禁止进入步骤5
5. 审查方案   自审清单 + 审查日志（修正项存入内存，暂不修改文件）
      ↓
6. 补搜资料   针对审查日志逐项搜索，产出状态标签存入内存
      ↓
7. 修正定稿   合并修正项+补搜结论+状态标签 → 更新文件/重新输出
```

## 轻量流程

适用于用户已告知技术方向、只需拆执行步骤的场景。跳过步骤 2（用户已确定方向，无需树状分解），跳过步骤 5-6（无需审查补搜）。

```
1. 分析问题 → 3. 快搜 → 4. 生成步骤列表（存内存） → 7. 输出分析结论+搜索结论+步骤列表
```

---

## 每步具体要求

### 1. 分析问题
*   先了解现状和现有约束。代码项目读源码（精确到文件名和行号），非代码项目列出关键参与方、资源或限制条件。凭记忆写出的分析不可靠。
*   **输出**（暂存上下文，待步骤 4 落盘）：明确定义每个问题的根本原因。

### 2. 解构问题
*   拆成独立分支，附加依赖标注（`【分支 X】名称（依赖 Y）` 或 `（独立）`）。
    > 复杂方案各部分有依赖关系。不标出来，排期时可能让被依赖的活排到后面，导致阻塞。树状结构帮你一眼看清"先做什么、什么可以并行"。
*   **完成后做两项检查：**

    **① 关键路径识别**
    看依赖图：哪个分支被最多其他分支依赖？在执行顺序里把它排最前。不需要额外标签，依赖关系本身已经说明优先级。

    **② 存量核查**
    对每个分支问：这个问题在现有系统/环境/文档里，是否已被（部分）解决？如果无法从步骤 1 的分析中确定，在分支标注里加 `[待核实]`。
*   **交互断点规则**（重要）：对于 `[待核实]` 分支，先不暂停。让步骤 3 的搜索优先尝试核实（查已有文档、源码、检索相关信息）。仅当步骤 1 的分析 + 步骤 3 的搜索仍无法确认时，才暂停并向用户提问。这样可以减少不必要的打断。
*   **输出**（暂存上下文）：分支列表（含依赖关系与状态标签）和超出本次范围的相关问题（作为边界声明的原始素材）。

### 3. 搜索资料
*   〔轻量流程〕：1-2 次核心搜索，结论内联于上下文。跳过本节其余所有内容。
*   **搜索范围**：搜索网络、项目内部文档、规范标准、实验记录均可作为信息来源。代码/技术类项目优先检索官方文档、GitHub 和英文资料；非代码项目优先检索行业报告、权威机构（.gov/.edu）和中文资料。
*   **优先核实**：对步骤 2 标记 `[待核实]` 的分支，首先尝试检索相关信息确认实际状态（搜索公网 + 检查用户已上传的项目文档）。确认后更新标签为 `[已确认存在]` 或 `[已确认不存在]`。仅当步骤 1+3 均无法确认时才暂停询问用户——"无法确认"的标准：无已上传项目文档可用，且公网搜索结果中个人/草根来源占比 > 80%，或存在明显事实冲突且无官方来源佐证。
*   每个分支至少检索一个核心信息点，代码项目优先英文关键词，非代码项目以中文为主。
*   **发散搜索**：核心技术搜索完成后，退一步做一次自由联想——这个方案落地后，谁会受影响？什么可能出错？哪些外部因素会干扰？用联想到的词再做一轮搜索。限 3 个方向，每个方向 1-2 次查询，不扩展更多。
*   **关键词降维策略**：直接搜索产品名无果时，按三步拆解重搜：功能拆解→组件映射→分层重搜。
*   **搜索结果暂存上下文**：按 `关键词 → 关键发现（一句话）→ 来源 URL [质量: 官方/社区/个人] [日期: YYYY]` 格式整理。终稿中必须保留关键词、来源 URL、质量标注三个字段。
*   **来源甄别**：整理时标注来源质量，按以下标准判定——
    *   官方/权威：域名 `.gov` `.edu`、项目官方文档、GitHub 仓库 Owner 账号、政府文件、上市公司财报
    *   社区/流行：Stack Overflow 得分 > 100、知乎/CSDN 获赞 > 500、GitHub Star > 1k、主流媒体高阅读量文章
    *   个人/草根：上述阈值以下或无法验证来源的个人博客、论坛帖、公众号文章
    > 非代码项目按对应领域的权威性类比判断（如制度文档 = 官方，行业头部 KOL = 社区，普通员工建议 = 个人）。超过 3 年的内容标注年份。

### 4. 写初稿
*   〔轻量流程〕：生成编号步骤列表，存于内存，不建文件。跳过本节其余所有内容。
*   **文件初始化**：有工程目录 → `.specify/memory/plans/{name}/plan.md`。无工程目录/非代码规划 → 当前工作目录创建 `{name}-plan.md`。若当前环境无文件系统（如纯对话界面），将完整文档暂存于上下文内存中，不写文件。（{name} 按上文「命名规则」取定）
*   **统一落盘**：读取步骤 1 的根因分析和步骤 3 的搜索结果，按以下顺序写入文件（或构建于内存中）：现状与根因 → 研究结论（含关键词+来源URL+质量标注） → 全图 → 路径详述 → 执行顺序 → 边界声明（读取步骤 2 的超出范围问题作为素材） → 测试策略。每个 ✅ 终点后简述验证方法。
*   **格式**：严格采用树状分支图，用段落叙述容易掩盖依赖关系和遗漏。
*   **验收标准**：每个分支走到 ✅ 终点。终点的描述应能让一个不熟悉该领域的同行明确判断完成与否。代码项目标注涉及的文件名，非代码项目标注关键约束或规格文档名称。必须包含「边界声明」章节，使用步骤 2 收集的超出范围问题作为素材写入。

### 5. 审查方案
在初稿（内存中）追加「审查日志」节，记录以下内容：
    > 审查的目的是逐项验证初稿的完整性，不是刻意找问题。每一项检查通过就直接标注通过，发现问题才记修正项。
*   **依赖顺序**：关键路径上的分支是否排在最前？独立分支是否可并发？
*   **遗漏检查**：有没有遗漏的边界情况？包括错误路径、向后兼容、性能退化、安全影响。回顾步骤 2 收集的「超出范围问题」，确认所有高风险项均已列入边界声明。
*   **可验证性**：每个 ✅ 终点的测试手段是否明确？
*   **补搜记录**：列出传递给步骤 6 的疑问点，格式 `[需补搜: 事实] 具体疑问` 或 `[需补搜: 策略] 具体疑问`。无则写「无」。
*   **修正项**：审查中发现的需修改的问题，记录在此。若无修正项，写明"经审查无发现问题，依据：[简述审查过程和判断理由]"。
*   汇总为三类：**确认项**（通过、无需改动）、**修正项**（已确定修改方案、待步骤 7 应用）、**补搜记录**（遗留疑问传给步骤 6）。

### 6. 补搜资料
*   读取审查日志中的补搜记录条目，逐项搜索。审查日志写「无」则跳过。
*   搜索范围同步骤 3：网络、内部文档、规范标准均可。
*   **区分补搜类型**：
    *   事实/数据缺失（如"Redis 7.0 maxmemory-policy 默认值"）→ 直接搜索。
    *   方法/策略缺失（如"如何设计缓存雪崩测试"）→ 走发散搜索逻辑（自由联想关键方向），而非降维策略。
*   直接搜索无果时走关键词降维。降维后仍连续 2 次无果，按以下逻辑树给标签：
    *   完全无信息、连推导都做不到 → `[无直接参考]`
    *   无外部参考，但基于通用原理可推导出具体步骤 → `[试验性方案]`
    > 两个标签由步骤 7 统一应用到对应分支。仅 `[试验性方案]` 需要在方案正文中加粗或高亮提示用户。本步骤不直接修改文件。

### 7. 修正定稿
*   〔轻量流程〕：输出三部分：①步骤 1 的分析结论 ②步骤 3 的搜索结论 ③步骤 4 暂存的编号步骤列表。无需建文件，无需升版本号。
*   **完整流程·修改阶段**：按以下顺序操作——
    1. 读取步骤 5 的修正项，逐条应用到草案正文的对应章节（重写并覆盖）。若无修正项，保留原文。
    2. 读取步骤 6 的补搜结论，更新研究结论中对应的发现。
    3. 读取步骤 6 的状态标签（`[无直接参考]` / `[试验性方案]`），应用到对应分支描述。
    4. 将审查日志追加到文档末尾（审查日志是执行过程的元数据，非方案正文。最终交付时可用「修订记录」或「审查说明」为题呈现）。研究结论中的每条发现确保保留关键词、来源 URL、来源质量标注三个字段。
*   **完整流程·输出阶段**：有文件系统 → 写入 `plan.md` 并升版本号。无文件系统（纯对话）→ 在回复中以 Markdown 代码块重新输出更新后的完整文档，按规则升版本号。

## 常见误区

*   **搜索为空 ≠ 不存在**：步骤 3 搜索空结果不意味着问题无解，应触发关键词降维。不要编造来源。
*   **审查不是走形式**：修正项不是绩效指标。审查日志全部通过是正常结果——有问题才记，没问题就记通过。
*   **降维不是万能钥匙**：降维后仍搜不到，老老实实标注 `[无直接参考]`。宁可承认信息缺口。
*   **别把分析当做事**：步骤 1/2 的目的是理解和分解问题，不是解决问题。常见错误是步骤 1 就开始构思解法，跳过了搜索，导致方案思路窄。

### 示例：数据库慢查询优化方案（缩略 · 代码项目）

```markdown
# 数据库慢查询优化方案
> 版本: v0.1.0 | 日期: 2026-06-02 | 状态: 草案

## 现状与根因
/api/users 接口 P99 延迟 3 秒，explain 显示全表扫描。根因：users 表缺少覆盖 WHERE + ORDER BY 的复合索引。

## 研究结论
### 关键发现 1 — PostgreSQL composite index
复合索引比单列索引减少 70-90% 扫描行数。
* 关键词: PostgreSQL composite index vs single column
* 来源: https://use-the-index-luke.com [质量: 知名技术博客] [日期: 2024]

## 全图
现在：users 表 20000 行，/api/users 接口 3 秒
  ├── 【分支 A】索引优化（独立）
  │     CREATE INDEX idx_users_status_created ON users(status, created_at DESC)
  │     ✅ EXPLAIN 显示 Index Scan，/api/users < 200ms · 验证：EXPLAIN ANALYZE
  └── 【分支 B】缓存层（依赖 A）
        Redis 缓存热点数据，TTL 5 分钟
        ✅ 缓存命中后 /api/users < 10ms · 验证：压测命中率

## 执行顺序
A → B

## 边界声明
- 写入性能优化（仅覆盖读查询）

## 测试策略
| 分支 | 验证点 | 方法 |
|:---|:---|:---|
| A | 索引生效 | EXPLAIN ANALYZE |
| B | 缓存命中 | 压测命中率 |

## 审查日志
* 确认项：索引字段顺序符合 WHERE + ORDER BY 最左前缀
* 修正项：分支 A 原计划只建单列索引，审查后改为复合索引
* 补搜记录：无
```

### 示例：新员工培训计划（缩略 · 非代码项目）

```markdown
# 新员工入职培训方案
> 版本: v0.1.0 | 日期: 2026-06-02 | 状态: 草案

## 现状与根因
近半年新员工上手周期平均 3 周，无系统化培训流程。

## 研究结论
### 关键发现 1 — 结构化入职培训
分阶段培训（文化→工具→业务）比一次性灌输留存率高 40%。
* 关键词: structured onboarding program best practices
* 来源: https://example.com/onboarding-study [质量: 官方] [日期: 2025]

## 全图
现在：新员工入职后各团队自行带教，标准不一
  ├── 【分支 A】文化培训（独立）
  │     参与方：HR + 部门负责人 → 公司历史、价值观、规章制度
  │     ✅ 新员工通过文化考核 · 验证：书面测试
  └── 【分支 B】工具与流程培训（依赖 A）
        参与方：IT + 直属上级 → 开发环境、权限申请、代码规范
```
