# 记忆分类标准

## 11 类定义

| # | 分类 | 含义 | 判定条件 |
|---|------|------|---------|
| 1 | 错误 | 被纠正的行为偏差 | 对话中出现纠正信号（"不对""错了""你应该……""以后不要……"）或 Claude 自己承认错误 |
| 2 | 决策 | 选择了 A 不选 B 的理由 | 两个以上方案被比较，最终选定一个并说明了理由 |
| 3 | 偏好 | 爸爸的表达偏好和好恶 | 爸爸说"以后……""不要……""我喜欢……"且针对 Claude 的行为/输出风格 |
| 4 | 模式 | 重复出现的行为规律 | 同类现象出现 ≥ 2 次，或从单次中可抽象为通用规律 |
| 5 | 排除 | 试过被否掉的方案 | 讨论了某方案但明确结论是"不这样做"、不采纳 |
| 6 | 不变量 | 永远成立的事实/约束 | 确认后不会改变的事实（技术限制、成本、架构边界） |
| 7 | 讨论 | 有结论的深度讨论 | 讨论涉及多次往返、方向修正，且最终有明确结论或产出 |
| 8 | 问题 | 发现的待修项/断链 | 发现 bug、链路断裂、配置缺失、引用错误——可追踪可关闭 |
| 9 | 知识 | 外部研究、技术事实 | 从外部查到的需要持久记住的信息（API 文档、工具机制、搜索结果） |
| 10 | 配置 | CC 设置变更 | settings.json / hooks / skills / commands / MCP 的修改记录 |
| 11 | 使用 | CC 使用踩坑和解决方案 | 遇到 CC 本身的操作问题或发现 CC 行为特性 |

## 判定决策树

```
提取到的信息是什么性质？
  ├── 被纠正的错误行为 → 错误
  ├── 选了 A 不选 B → 决策
  ├── 爸爸的喜好/风格要求 → 偏好
  ├── 多次出现的规律 → 模式
  ├── 明确不做的方案 → 排除
  ├── 永不改变的事实 → 不变量
  ├── 深度讨论及结论 → 讨论
  ├── 发现待修的东西 → 问题
  ├── 外部查到的信息 → 知识
  ├── CC 配置修改 → 配置
  └── CC 使用踩坑 → 使用
```

## 一条信息可归属多个分类

同一个事件可产出多条记忆，分属不同分类。共享 `事件: evt-xxx`。不同分类间用 `关联:` 双向链接。

## 每种分类的提取信号

在去噪后的对话流中，扫描以下信号决定提取什么：

| # | 分类 | 对话中检测什么 | 提取规则 |
|---|------|--------------|---------|
| 1 | 错误 | 爸爸的纠正语句（"不对""错了""你应该……"）、Claude 的自我纠正（"抱歉""我搞错了"） | 提取被纠的偏差行为 + 正确做法 + 违反的规则 |
| 2 | 决策 | 两个以上方案被讨论、爸爸说"这样吧""决定了""就这个" | 提取选项对比 + 最终选择 + 选择理由 |
| 3 | 偏好 | 爸爸说"以后""不要""我喜欢""讨厌"、对 Claude 风格的反馈 | 提取偏好的具体内容 + 是否已有类似偏好 |
| 4 | 模式 | 同类行为/错误在本会话或跨会话 ≥ 2 次 | 提取模式描述 + 已出现次数 + 正/负面 |
| 5 | 排除 | 讨论了某方案但明确结论是"不做了""不选这个" | 提取被否的方案 + 否决理由 |
| 6 | 不变量 | 确认后不会改的事实（"永远""不可能""上限是"） | 提取事实 + 确认来源 |
| 7 | 讨论 | 多次往返讨论 + 有明确最终结论 | 提取讨论主题 + 结论 + 关键转折点 |
| 8 | 问题 | 发现 bug、断链、配置缺失（"为什么…没有了""缺了…"） | 提取问题描述 + 影响面 + 待修状态 |
| 9 | 知识 | WebSearch/WebFetch 调用、外部文档查阅、搜索结果 | 提取查到的关键事实 + 来源 URL + 日期 |
| 10 | 配置 | Edit/Write 操作 `settings.json` `hooks/` `skills/` `commands/` | 提取改了什么字段 + 为什么改 + 改前改后对比 |
| 11 | 使用 | 发现 CC 行为特性、踩坑、CC 版本变化 | 提取发现 + 影响 + 解决方案（如有） |

**提取范围**：一次扫描覆盖以上 11 类信号。不要求每类都有产出——该对话有什么就提什么，没有就不提。

**提取粒度**：一个事件块内，同一分类下若多次出现同类信号 → 合并为一条（如：同一个事件里爸爸纠正了 3 次"跳 skill" → 一条"错误"记忆）。不同事件块 → 独立提取。

## 示例

```
对话中 Claude 说"中转站机制应该废弃"
  → 决策: 舍弃中转站改用 JSONL 回炉
  → 问题: 中转站碎片产生链路断裂（触发此决策的根因）
  → 两个文件，同一事件 ID，双向关联
```

## 易混淆分类对比

以下三组分类容易误判，特别注意：

### 偏好 vs 配置 vs 使用

| 场景 | 正确分类 | 判断依据 |
|------|---------|---------|
| 爸爸说"以后方案别列大纲" | **偏好** | 爸爸对 Claude 行为/输出风格的喜好 |
| 修改了 settings.json 加了一个 hook | **配置** | 对 CC 设置文件的实质性修改 |
| 发现 CC 的 `/compact` 在 50% 就该建议 | **使用** | 对 CC 行为的观察/踩坑，不涉及修改配置文件 |
| 爸爸要求"结果直接说别铺垫" | **偏好** | 爸爸的表达风格要求 |
| 安装了 find-skills 技能 | **配置** | skills 的安装/卸载/修改 |
| 发现 skill 装完后需要重启会话才生效 | **使用** | CC 行为特性的发现 |

判断优先级：先看是否涉及**爸爸的喜好**（→偏好），再看是否涉及**文件修改**（→配置），都不满足但和 CC 行为相关（→使用）。

### 错误 vs 问题

| 场景 | 正确分类 | 判断依据 |
|------|---------|---------|
| Claude 跳了 skill 被爸爸指出 | **错误** | 行为偏差，已被纠正 |
| 发现 Harness 层 hook 全部缺失 | **问题** | 发现的待修项，不是行为偏差 |
| Claude 说"已清理完毕"但没验证 | **错误** | 违反工具操作律 |
| 依赖地图引用旧目录结构 | **问题** | 文件内容过时，待修复 |

### 知识 vs 不变量

| 场景 | 正确分类 | 判断依据 |
|------|---------|---------|
| DeepSeek 缓存命中 ¥0.1/M token | **知识** | 外部查到的信息，可能随时间变化 |
| deep-research 约 300 万 token/次 | **不变量** | 确认后不会变的技术事实 |
| CC 2.1.163 新增 additionalContext | **知识** | 版本更新，后续版本可能再变 |
